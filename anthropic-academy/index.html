导言 · WHY THIS GUIDE

轨道 1 · FOUNDATION · 2 门课 · 约 1.5h

入门基础 — 地图和心智模型

两门课构成 Anthropic Academy 的「地图」:一门告诉你 Claude 桌面端能干嘛(产品全景),一门告诉你 LLM 底层怎么思考(避免被幻觉骗)。任何角色都该先过这两门,各 45 分钟。

01 · FoundationClaude 101 — 第一次用 Claude 就上手

课程 01

Claude 101

Claude 101 — Daily-work tasks, core features, advanced resources
FOUNDATION ~45 min 5 模块 / 13 课
讲什么
把 Claude 产品全家桶过一遍:Chat、Cowork、Code、Projects、Artifacts、Skills、Connectors、Enterprise Search、Research Mode。重点不是教你写 prompt,是告诉你「桌面端这堆按钮分别是干啥的」。
给谁
所有没用过 Claude 桌面端的人。产品经理、运营、市场、销售、HR、写作者、第一次接触 Claude 的开发者都适合。
章节
  • Meet Claude — 第一次对话 / 怎么拿到更好的结果 / 桌面端三件套(Chat/Cowork/Code)
  • Organizing — Projects 项目 / Artifacts 工件 / Skills 能力包
  • Expanding — Connectors 工具连接 / Enterprise Search 企业搜索 / Research Mode 深度调研
  • Putting it together — 按角色看用例 / 其他工作方式
  • Conclusion + 完成证书
我的判断
值得花 45 分钟。这是唯一让你看清 Claude 全景的官方课——其他课都是聚焦在某个具体能力(Code/Cowork/Skills),只有这门告诉你「这些能力如何组成一个产品矩阵」。
跳过条件
你已经在日常用 Claude 桌面端、知道 Projects/Artifacts/Connectors 各自的位置,就可以跳过。
→ anthropic.skilljar.com/claude-101

02 · FoundationAI 能力与局限 — 知道 LLM 在想什么,才不会被骗

课程 02

AI 能力与局限

AI Capabilities and Limitations — Mental model of how generative AI works
FOUNDATION ~45 min 7 模块
讲什么
给你一套 LLM 心智模型,围绕 4 个性质展开:下一词元预测(Next Token Prediction)、知识(Knowledge)、工作记忆(Working Memory)、可操控性(Steerability)。每个性质都配「Try it out」实操。最后一章 When Properties Collide 教你看穿幻觉、低估、过度自信的根源。
给谁
所有人,尤其是「会用 Claude 但被它骗过」「想知道为什么有时它一本正经胡说八道」的人。商业决策者、合规、法务、记者特别合适。
章节
  • Getting Started — 课程介绍
  • What We Mean by AI / How AI Gets Its Character — 定义与性格形成
  • Next Token Prediction — 核心机制 + 实操
  • Knowledge — 它「知道什么」+ 实操
  • Working Memory — 上下文窗口 + 实操
  • Steerability — 怎么把它「拽」到你要的方向 + 实操
  • Putting It All Together — When Properties Collide / Next Steps / Quiz
我的判断
2026 年 Anthropic 新出的中立独立课(不绑特定产品),少有的从底层原理讲起的课。45 分钟拿到「不被 LLM 骗」的免疫力,ROI 极高。
跳过条件
你已经读过 Karpathy 的 LLM 介绍、或写过 Transformer 推理代码,就可以跳过——这门课是给「会用但不懂内部」的人补底层。
→ anthropic.skilljar.com/ai-capabilities-and-limitations
尖锐共识 · SHARP TAKE

这两门入门课互补不重复:Claude 101 教「Claude 能干嘛」,AI Capabilities and Limitations 教「LLM 在想什么」。前者是产品功能图,后者是大脑解剖图。两门都过完,后续任何课你都能更快理解——因为底层心智模型已经搭好了。1.5 小时投入,后续每门课至少节省 30% 时间,这是 Academy 里 ROI 最高的组合。

轨道 2 · AI FLUENCY · 6 门课 · 约 3–4h

AI 素养 — 4D 框架是底层,其他 5 门是人群细分

由 Anthropic 与学术合作伙伴 Prof. Joseph Feller、Prof. Rick Dakan 共同开发,核心是 4D 框架(Delegation / Description / Discernment / Diligence)。一门主课讲框架本身,其余 5 门是给特定人群(教师、学生、NGO、小企业、培训师)的应用版。

★ 看这个轨道之前先看这里 4D 框架是其他 5 门的基础——直接跳过去看人群版会迷失。建议顺序:先看 03(Framework & Foundations),再选一个跟你身份对应的人群版。把 6 门全刷一遍是浪费时间,内容重叠度 60%+。

03 · Foundation · 主课AI Fluency: Framework & Foundations — 跟 AI 打交道的 4D 方法论

课程 03 · ★ 主课

AI 素养 · 4D 框架

AI Fluency: Framework & Foundations
FOUNDATION ~1.1h 8 模块
讲什么
提出 4D 框架:Delegation(委派)/ Description(描述)/ Discernment(辨别)/ Diligence(勤勉)。每个 D 一节课,核心是两个循环——「Description-Discernment 循环」(描述→评估反复迭代)和「Delegation-Diligence 循环」(委派→负责任地监督)。
给谁
所有人。这门课不教 prompt 技巧,教的是「人和 AI 怎么分工」的元方法论。
章节
  • Introduction to AI Fluency
  • The AI Fluency Framework — 4D 模型概览
  • Deep Dive 1: Generative AI Fundamentals
  • Delegation — 怎么把任务交给 AI
  • Description — 怎么把想要的东西讲清楚
  • Discernment — 怎么评估 AI 的输出
  • Diligence — 怎么对结果负责
  • Conclusion & Certificate
我的判断
学术派课程,稍嫌啰嗦,但 4D 框架确实是个有用的语言工具——之后你和团队讨论「怎么用 AI」时,有一套共同词汇。如果你只刷一门 AI Fluency,选这门。
跳过条件
你已经能流畅区分「prompt 设计 / 输出校验 / 信任校准 / 责任归属」四个维度,就可以跳过——这门课其实就是把这四个东西命名为 4D。
→ anthropic.skilljar.com/ai-fluency-framework-foundations

核心 1 · METHODOLOGY4D 框架 — 跟 AI 打交道的元方法论

整个 AI Fluency 轨道(6 门课)都在讲这一个东西。本质上,它把「人和 AI 协作」拆成 4 个动作,让你能精确说出「哪一步出了问题」。

D
Delegation · 委派

把任务交给 AI

判断这件事该不该让 AI 做。识别哪些任务适合 AI(重复、可结构化、容错高),哪些必须人做(高风险决策、原创判断、需要责任归属)。
D
Description · 描述

把想要的讲清楚

prompt 工程的本质——给足上下文、明确产出格式、举例。说不清楚的任务,AI 做不好不是它的错。
D
Discernment · 辨别

评估 AI 的输出

看到结果不要照单全收——它哪里对、哪里编、哪里漏。AI 越流畅,越容易让你跳过这一步。
D
Diligence · 勤勉

对结果负责

输出最终归你署名。把关、保密、合规、伦理是你的事。AI 不会替你坐牢。

4D 不是孤立的 4 个动作,而是两个循环:

  • Description-Discernment 循环 — 写 prompt → 看结果 → 改 prompt → 再看,直到对。这是具体任务层的迭代。
  • Delegation-Diligence 循环 — 选任务 → 让 AI 做 → 监督 → 调整任务边界。这是责任和分工层的循环。
★ 这套框架的用处 不在于它说了多么新的东西(每个老 prompt 工程师都知道这些)。在于给团队提供一套共同语言——下次说「这个 prompt 不行」,你可以精确说「是 Description 没讲清,还是 Discernment 没做」。一个团队 prompt 协作,有 4D 词汇 vs 没有,效率差好几倍。

核心 2 · AGENT MECHANICSAgentic Loop — 编码代理跟聊天助手的本质区别

Claude Code 101 / Claude Code in Action 反复强调:Claude Code 不是「能写代码的聊天助手」,是 Agent(代理)。Agent 跑的是一个三步循环:

┌─────────────────────────────────────────────────┐
│   ① Gather Context  →  ② Take Action  →  ③ Verify Work   │
│        (收集上下文)        (采取行动)         (验证结果)        │
└─────────────────────────────────────────────────┘
                        ↑                                       │
                        └───────── loop ─────────────────────┘
  • ① Gather Context — 读文件、列目录、跑 grep、查 git log、看 CLAUDE.md / Skills。它在动手前先看清现场。
  • ② Take Action — 编辑文件、跑命令、调 MCP 工具、提 PR。真正的动作发生在这里。
  • ③ Verify Work — 跑测试、看输出、对比 diff、检查 lint。它自己审自己的活。

循环回到 ① 是因为 ③ 的结果改变了上下文(测试失败了 / 文件变了),需要重新收集情况再决定下一步。这个循环就是「Agent」的定义——能自主感知-行动-验证-再行动的系统。

尖锐共识 · SHARP TAKE

大部分人用 Claude Code 出问题,根源是没意识到这是 Agent 而不是 Chat。Chat 习惯是「问问题→拿答案」,Agent 模式是「描述任务→看它跑→在关键节点确认」。你跟 Agent 互动的最佳姿势是「监工」,不是「点菜」。

核心 3 · WORKFLOWExplore → Plan → Code → Commit — Claude Code 哲学

这是 Claude Code 推荐的四阶段工作流。Claude Code 101 反复讲这个,因为它把 Agentic Loop 落到了具体可操作的步骤。

阶段该做什么该用的工具
① Explore
勘察		 读相关文件、看现有代码风格、查 git 历史、识别约束。不写一行代码。 Read / Glob / Grep / git log
② Plan
规划		 用 Plan Mode(只读模式)让 Claude 先输出实施计划。你审一遍,定下范围。 Plan Mode + 用户确认
③ Code
编码		 按 Plan 执行。可以用 Auto-Accept(自动接受编辑)加速,但要在 Plan 已确认的前提下。 Edit / Write / Bash
④ Commit
提交		 跑测试 + lint + 类型检查 → 写 commit message → 提交。让 Agent 自己审一遍 diff 后再提。 Bash(npm test / git commit)

三种 Approval Mode 用在哪一阶段

  • Plan Mode(只读) — 用在 Explore + Plan。Claude 只能读、不能改,你审完再让它进入 Code 阶段。
  • Manual(每步确认) — 用在 第一次接触陌生代码库。每个编辑都要你点确认,慢但安全。
  • Auto-Accept(自动接受) — 用在 Plan 已确认 + 你信任 Claude 在这个范围内不会乱来。最快,但只有 Plan 充分时才用。
★ 常见错误 跳过 Explore + Plan,直接让 Claude「写一个 X」——结果它没看上下文,写出风格不一致、跟现有代码冲突的版本。Explore 看似浪费时间,实际是节省时间。

核心 4 · DISAMBIGUATIONSkills / Subagents / Hooks / MCP — 4 个东西怎么分

Claude Code 有 4 种主要的「自定义机制」,新人最容易混。区分清楚这 4 个,你才能在合适的场景选合适的工具。

机制 本质 什么时候用 怎么用
Skill 装着 SKILL.md 的文件夹,教 Claude 「怎么做某类任务」 有可重复工作流,3 次以上要解释同一个流程 写 SKILL.md + description,Claude 看到匹配场景自动加载
Subagent 带独立上下文窗口的小代理,主代理派任务给它 任务能切片、要并行、想保持主上下文干净 /agents 命令创建,主代理调用
Hook 事件触发的脚本(PreToolUse / PostToolUse / UserPromptSubmit 等) 要在每次特定动作前后强制跑某个检查 / 注入逻辑 settings.json 里配,bash 脚本执行
MCP 外部协议,让 Claude 接入外部系统(数据库 / API / 内部服务) 要让 Claude 读你公司的私有数据 / 跑外部工具 搭 MCP Server(Python SDK),Claude 连上后能调用

关键判断:遇到一个需求该用哪个?

  • 「希望 Claude 知道怎么做某件事」(写邮件、跑测试、转 PDF) → Skill
  • 「希望 Claude派一个分身去做某件事,主线不被打断」 → Subagent
  • 「希望每次特定动作前后都强制做某个检查」(commit 前跑 lint、edit 后自动 format) → Hook
  • 「希望 Claude 能访问外部系统」(查 Linear、读 PostgreSQL、调内部 API) → MCP
★ 组合用 它们不是互斥的。一个完整的高级用法:写一个 Skill 教 Claude「怎么发布到 Production」,Skill 里调用一个 MCP server 访问 deploy 系统,Skill 在 Subagent 里跑(主代理保持干净),整个过程靠 Hook 在 commit 前强制做安全扫描。这 4 个机制各管一段,叠加起来才是 Claude Code 的真正威力。

核心 5 · PRODUCTIONPrompt Caching + Extended Thinking — API 工程级技巧

这两个是 Building with Claude API / Bedrock / Vertex 三门课都讲的生产级优化。不懂这两个,你的 API 账单会比应该贵 3–10 倍,响应也慢得多。

Prompt Caching · 五条规则

原理:把 prompt 里重复的前缀部分缓存住,后续请求只算「新增部分」的 token,价格降到原价的 10%。规则:

  1. 缓存的是前缀 — 必须从消息开头的 system / 第一条 user message 开始,中间不能断。改后面的内容不破坏前面的缓存。
  2. 最小 1024 tokens(Claude Sonnet 上),小于这个不缓存。
  3. 5 分钟 TTL — 写完 5 分钟没人用就过期,下次重新缓存。可付费延长到 1 小时。
  4. 缓存创建有写入费(原价的 1.25 倍),命中后才便宜。一次性请求别开缓存,反而贵。
  5. 用 cache_control 字段显式标记要缓存的内容块。最多 4 个 breakpoint。

什么时候开:同一段长 system prompt / 工具定义 / 文档检索片段会在 5 分钟内被多次调用——开。一次性查询、prompt 每次都不一样——不开。

Extended Thinking · 让 Claude 想得更深

这是 Claude 在回答前先做内部「思考」的模式,适合复杂推理任务。课程教的关键点:

  • 用 thinking 参数开启,设 budget_tokens 控制思考长度。
  • 思考内容会消耗 token,但输出质量在数学 / 多步推理 / 代码审查类任务上显著提升。
  • 思考内容不直接展示给用户(默认),但你可以读 thinking 块拿到。
  • 对简单任务别开——浪费 token 又慢。

核心 6 · ARCHITECTUREAgent Workflow 四模式 + MCP 三原语

建多 agent 系统时反复出现的两个分类。Building with Claude API 第 10 章 + 云课的 Agents 章节都讲这个。

Agent Workflow 四种模式

模式结构什么时候用
Parallelization
并行		 主任务拆成 N 个独立子任务,N 个 agent 同时跑,结果汇总 子任务彼此独立(分析 N 个文件、N 个产品对比、N 份候选人简历)
Chaining
链式		 第一个 agent 输出 → 第二个 agent 输入 → 第三个 ... 串行流水线 每一步依赖上一步结果(草稿 → 翻译 → 校对 → 排版)
Routing
路由		 一个「分流 agent」根据输入类型把任务派给不同 specialist agent 输入有多种类别(客服分流到「退款」「技术」「投诉」三个专用 agent)
Agent
自主代理		 单 agent 在循环里自己决定下一步用哪个工具,直到任务完成 路径不可预测的开放任务(写一个新功能、调试一个 bug)
★ 怎么选 Anthropic 的建议是能用 Workflow 就别上 Agent——前三种(Parallelization / Chaining / Routing)是结构化的,行为可预测、成本可控、容易调试。Agent 模式灵活但贵、慢、难追踪。先看你的任务能不能拆成前三种,只有真的拆不动才上 Agent。

MCP 三大原语

MCP 协议让 Claude 跟外部系统对接。三种交互类型,各自的「控制权」不一样:

原语谁控制典型场景
Tools
工具		 Model-controlled — Claude 自己决定何时调用 查数据库、调 API、跑命令、发邮件
Resources
资源		 Application-controlled — 应用主动提供,Claude 引用 提供文件内容、配置、文档、上下文数据
Prompts
提示		 User-controlled — 用户从模板里选 预设的常用任务模板(/translate /summarize /review)

区分这三种的关键:「谁决定何时使用」。Tools 是 LLM 主动调,Resources 是应用塞给 LLM,Prompts 是用户从菜单选。设计 MCP server 时,先想清楚每个能力归哪一类——错位会让 Claude 用不上你的服务。

尖锐共识 · SHARP TAKE

这 6 个核心概念是看完 18 门课的真正收益。剩下的内容(具体 API 调用、特定云接入流程、人群版的应用场景)在你需要时查文档就够,不必死记。把这 6 个概念吃透,你就有了整个 Anthropic 生态的心智模型——以后看任何新课、新文档、新博客,都能立刻判断它在讲哪个核心机制的延伸。

术语速查 — 看课和看文档都用得上

25 个 Anthropic 生态里反复出现的术语,中英对照 + 一句话定义。看任何课、读任何文档时,这一页可以当 reference。

Agent
代理 / 智能体
能自主跑「感知-行动-验证」循环的 AI 系统。区别于 Chat(被动应答)。Claude Code 是 Agent 不是 Chat。
Agentic Loop
代理循环
Gather Context → Take Action → Verify Work 三步循环。Agent 的核心机制。
Skill / SKILL.md
能力包 / 能力说明书
装着 SKILL.md 的文件夹,教 Claude 怎么做某一类任务。Claude 自动识别场景加载。
Subagent
子代理
带独立上下文窗口的小代理,主代理派任务给它。用 /agents 命令创建。
Hook
钩子
特定事件触发的脚本(PreToolUse / PostToolUse / UserPromptSubmit 等)。在 settings.json 里配。
MCP · Model Context Protocol
模型上下文协议
Anthropic 开放标准,让 Claude 跟外部系统对接。三原语:Tool / Resource / Prompt。
CLAUDE.md
项目说明书
放在项目根的 Markdown 文件,Claude Code 启动时自动读,作为「项目级 system prompt」。
Plan Mode
规划模式
Claude Code 的只读模式——只能看不能改,产出实施计划让你审。用在 Explore + Plan 阶段。
Auto-Accept
自动接受
Claude Code 的快速模式——自动接受所有编辑,不打断你。Plan 已确认时才用。
Approval Mode
批准模式
Claude Code 的三种交互模式:Plan / Manual / Auto-Accept,控制 Claude 行动前是否要确认。
Prompt Caching
提示缓存
把重复的 prompt 前缀缓存,后续请求只算新增部分 token,降到原价 10%。5 分钟 TTL。
Extended Thinking
深度思考
Claude 在回答前先做内部「思考」的模式,适合复杂推理。用 budget_tokens 控制思考长度。
Tool Use
工具调用
Claude 通过返回 tool_use block 让客户端执行某个函数,拿到结果后继续生成。Agent 的基础能力。
System Prompt
系统提示
最前面的指令,告诉 Claude 它是谁、做什么、有什么约束。区别于用户消息。
RAG · Retrieval Augmented Generation
检索增强生成
先检索相关文档片段,再让 Claude 基于这些片段回答。让 Claude 能用你的私有数据。
Streaming
流式响应
API 边生成边返回,而不是等全部完成再返回。改善用户体验。
Sampling (MCP)
采样(MCP 概念)
MCP server 反向请求 client 调 LLM 的机制。绕开 server 自己持有 API key。
Roots (MCP)
根路径(MCP 概念)
MCP 里定义文件访问权限边界的机制——告诉 client 哪些路径可读。
StreamableHTTP
可流式 HTTP 传输
MCP 推荐的生产级传输层,取代旧的 SSE。支持水平扩展、无状态。
Cowork
协同模式
Claude 桌面端的一个工作模式——直接操作你本地的文件、应用。区别于 Chat 模式。
Plugin
插件
Claude Code / Cowork 的能力扩展包,通常打包 Skills + Hooks + MCP servers + Commands。
Connector
连接器
Claude 桌面端预置的工具连接(Gmail / Calendar / Linear 等),无需自己搭 MCP。
4D Framework
4D 框架
Delegation / Description / Discernment / Diligence,人和 AI 协作的元方法论。
Workflow vs Agent
工作流与代理
Workflow 是预定义路径(并行/链式/路由),Agent 是自主循环。能用 Workflow 就别用 Agent。
Computer Use
计算机操作
Claude 通过截图 + 鼠标键盘事件操作整个桌面 UI 的能力。当前还是 Beta。
