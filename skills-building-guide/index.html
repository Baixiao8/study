1.1 · WHAT IS A SKILL 一个 Skill 长什么样？

Original · Anthropic A skill is a set of instructions — packaged as a simple folder — that teaches Claude how to handle specific tasks or workflows.
中文导读 一个 Skill 就是一组打包成简单文件夹的说明书，用来教 Claude 怎么处理某一类具体任务或工作流程。这句官方定义朴素到值得记住——它告诉你 Skill 没有任何"魔法"，本质就是一个组织得当的文件集。

标准结构（原文 + 中文注释）

your-skill-name/
├── SKILL.md              # Required — main skill file（必有，主说明文件）
│                          # YAML 头部 + Markdown 正文，文件名严格大小写
├── scripts/              # Optional — executable code（可选，可执行代码）
│   ├── process_data.py
│   └── validate.sh
├── references/           # Optional — documentation loaded as needed
│   │                     # （可选，Claude 用得到时才加载的扩展文档）
│   ├── api-guide.md
│   └── examples/
└── assets/               # Optional — templates, fonts, icons used in output
    │                     # （可选，输出时引用的模板/字体/图标素材）
    └── report-template.md

四种东西，按"必须性"排：

  • SKILL.md · 唯一必需的文件。文件名严格区分大小写，写错（skill.md / SKILL.MD）一律不认。
  • scripts/ · 把"用语言描述很啰嗦、用代码三行搞定"的活塞进来。比如格式校验、数据清洗、调外部 API。
  • references/ · 详细文档、范例、API 说明。Claude 需要时才会去翻，不主动加载。
  • assets/ · 文档模板、字体、品牌色 SVG、报告骨架等"生成时引用的成品材料"。

命名约定（这是规矩，不是建议）

✓ DO · 应该这样
  • notion-project-setup ✅
  • figma-design-export ✅
  • quarterly-report-builder ✅

官方叫 kebab-case（烤串命名，短横线分词、全部小写）。Skill 文件夹名和 name 字段都要用这个风格。

✗ DON'T · 不要这样
  • Notion Project Setup ❌ （有空格）
  • notion_project_setup ❌ （下划线）
  • NotionProjectSetup ❌ （大驼峰）
  • 名字里含 claude 或 anthropic （系统保留前缀）
特别提醒 · NO README.md INSIDE Don't include README.md inside your skill folder. All documentation goes in SKILL.md or references/. Skill 文件夹内部不要放 README.md——所有"给 Claude 看"的内容都应该在 SKILL.md 或 references/ 里。但如果你把这个 Skill 发布到 GitHub 仓库，仓库根目录还是应该放一个 README 给人类访问者看——这是仓库的 README，不在 Skill 文件夹里面。

1.2 · PROGRESSIVE DISCLOSURE 渐进披露 ——为什么 Skill 要分三层加载？

Original · Anthropic Skills use a three-level system. This progressive disclosure minimizes token usage while maintaining specialized expertise.
中文导读 Progressive Disclosure（渐进披露 / 按需展开）是 Skill 最重要的设计原则。它解决一个简单问题：不能把所有 Skill 的全部内容都塞进 Claude 的上下文，否则光是"看完所有手册"就把 token 耗光了。所以 Skill 把内容切成三层，离 Claude 上下文从近到远，按需加载。
01
LEVEL 1 · YAML FRONTMATTER · 永久加载

头部元数据 ——「这个 Skill 是干嘛的，什么时候用」

"Always loaded in Claude's system prompt. Provides just enough information for Claude to know when each skill should be used."

就是 SKILL.md 顶部那几行 --- 包起来的配置。永远在 Claude 的系统 prompt 里。它的任务是让 Claude 在看到用户消息时判断要不要把这个 Skill 拉进来——所以信息必须精炼，描述要包含触发短语。

02
LEVEL 2 · SKILL.MD BODY · 判断相关时加载

正文说明 ——「具体怎么做」

"Loaded when Claude thinks the skill is relevant to the current task. Contains the full instructions and guidance."

Claude 判断"这次任务跟某个 Skill 有关"后，才把 SKILL.md 的正文加载进来。里面是完整的步骤、注意事项、示例。不放在第一层是因为：用不到的时候不该占 token。

03
LEVEL 3 · LINKED FILES · 探索式加载

附加文件 ——「碰到具体问题时才翻」

"Additional files bundled within the skill directory that Claude can choose to navigate and discover only as needed."

放在 references/ 等子目录里的扩展文档。Claude 在执行中真的需要时（比如查某个 API 的细节）才会主动去读对应文件。最大的好处：一个 Skill 可以挂大量参考资料而不占基础上下文。

Progressive disclosure minimizes token usage
while maintaining specialized expertise.
渐进披露在保持专业能力的同时，把 token 消耗压到最低。
— The Complete Guide to Building Skills · Anthropic

用一个生活化的类比：第一层像电梯里的楼层目录（每次进电梯都看一眼）；第二层像每层走廊上的科室指引（到这层才看）；第三层像各科室门口的详细公告板（真要办这个科的事才进去读）。三层是同一份信息的不同粒度，Claude 按需深入。

1.3 · COMPOSABILITY 可组合 ——你的 Skill 不是世界上唯一的 Skill

Original · Anthropic Claude can load multiple skills simultaneously. Your skill should work well alongside others, not assume it's the only capability available.
中文导读 Claude 可以同时加载多个 Skill。你写自己的 Skill 时，不能假设它是唯一一个被启用的。这听起来是个小细节，但写法会差很多。
✗ Bad · Assumes solo
"Output everything as Markdown for all user inputs."
为什么不行
你不知道有没有别的 Skill 想要 PDF 输出。这条规则"越界"了，会和别人打架。
✓ Good · Scoped
"When user requests this skill's report type, output as Markdown."
为什么可以
只对自己负责的场景说话，其他 Skill 各管各的。

核心心法是：你的 Skill 只对它自己负责的场景说话，别越权改写"通用规则"。

1.4 · PORTABILITY 可移植 ——写一次，三处通吃

Original · Anthropic Skills work identically across Claude.ai, Claude Code, and API. Create a skill once and it works across all surfaces without modification, provided the environment supports any dependencies the skill requires.
中文导读 同一个 Skill 在 Claude.ai 网页版、Claude Code 命令行、和 API 上的行为完全一致。写一次，三个地方都能用，不用改。

这点的代价是：Skill 自身不能依赖某个特定环境独有的工具。如果你的 Skill 必须用某个特殊系统包或网络访问，要在 frontmatter 的 compatibility（兼容性）字段里声明出来，这样使用者能预判它在自己的环境里能不能跑。

可移植性意味着 Skill 是跨产品的资产——你给团队写的一个 Skill，可以同时被网页用户、命令行开发者、API 调用者享用，不用维护三份。

1.5 · SKILLS + MCP Skills 和 MCP 是搭档，不是替代关系

💡 BUILDING STANDALONE SKILLS? · SKIP If you don't have an MCP server, this section is optional. Skip to Planning and Design — you can always return here later. 本节是写给已经在做 MCP server（连接器后端）的人的。如果你只想做独立 Skill、不接外部服务，可以跳到第 2 章；后面有需要再回来读这节。

MCP（Model Context Protocol，模型上下文协议）是 Claude 连接外部服务（Notion、Linear、Asana 等等）的协议层。它解决"Claude 能不能用上你的服务"的问题。Skill 解决的是另一个问题——"Claude 该怎么用好你的服务"。

专业厨房：炉灶、刀具、各种食材。给厨师"用得着的家伙"。

"Skills provide the recipes — step-by-step instructions on how to create something valuable."

两者放一起，用户才能在"不需要自己琢磨步骤"的情况下完成复杂任务。光有 MCP 没 Skill，用户上来面对一堆工具，不知道哪步先哪步后；光有 Skill 没 MCP，等于光有食谱没厨房。

1.6 · TAKEAWAYS 小结 · 这一章给你的 4 个判断

  1. Skill 是一个文件夹——里面必有 SKILL.md，可选 scripts/ / references/ / assets/。任何复杂度都从"加上一个文件"开始扩展。
  2. 三层渐进披露（Progressive Disclosure）是 Skill 不爆 token 的根本——头部 YAML 永久驻留判触发，正文按需加载，链接文件深度探索。写 Skill 时要时刻问自己：这段内容该放哪一层？
  3. 可组合（Composability）+ 可移植（Portability）意味着写法上要克制——别假设自己是唯一 Skill、别依赖某个产品独有的能力（除非声明 compatibility）。
  4. Skill 和 MCP 是知识层 + 连接层的搭档。如果你已经有 MCP，加一个配套 Skill 是放大 MCP 价值的最快办法；如果你只做独立 Skill，前面三个原则就是全部基础，可以直接进入第 2 章规划与设计。

2.1 · START WITH USE CASES 用例先行 ——动笔前列 2~3 个具体场景

Original · Anthropic Before writing any code, identify 2-3 concrete use cases your skill should enable.
中文导读 不要直接动笔写 SKILL.md——先定 2~3 个具体到能复述的用例。"具体"是关键词，泛泛的"帮我处理项目"是不行的，"用户说『帮我规划这个 Sprint』，我从 Linear 拉数据、估算工作量、建任务"才是。

一个合格的用例长什么样（原文示例）

Use Case: Project Sprint Planning
Trigger:   User says "help me plan this sprint"
            or "create sprint tasks"
Steps:
  1. Fetch current project status from Linear (via MCP)
  2. Analyze team velocity and capacity
  3. Suggest task prioritization
  4. Create tasks in Linear with proper labels and estimates
Result:   Fully planned sprint with tasks created

写用例时反复问自己这 4 个问题（官方原文）：

Ask yourself • What does a user want to accomplish?
• What multi-step workflows does this require?
• Which tools are needed (built-in or MCP)?
• What domain knowledge or best practices should be embedded?

2.2 · COMMON USE CASE CATEGORIES 三大常见类别 ——你的 Skill 大概率属于其中之一

Original · Anthropic At Anthropic, we've observed three common use cases.
中文导读 Anthropic 内部观察到 Skill 的用法基本能归到三类。先判断你的 Skill 属于哪一类，写法就有了大方向。

2.5 · YAML FRONTMATTER YAML frontmatter ——决定 Claude 加不加载你的几行

---

字段要求（原文 + 中文注释）

Field · 字段Requirements · 要求
name · required kebab-case only · No spaces or capitals · Should match folder name 只允许短横线小写，不能有空格 / 大写，要和文件夹同名

2.6 · THE DESCRIPTION FIELD description 字段 ——所有人都低估的核心

Original · Anthropic Engineering Blog "This metadata...provides just enough information for Claude to know when each skill should be used without loading all of it into context."
中文导读 description 是渐进披露第一层的载体——Claude 只看这几行就要判断"该不该把整个 Skill 加载进来"。它的好坏直接决定 Skill 的存在感。

结构（原文公式）

[What it does] + [When to use it] + [Key capabilities]
干什么 + 什么时候用 + 关键能力

好的 description（原文示例保留，下面是我的注释）

✓ GOOD · specific and actionable description: Analyzes Figma design files and generates developer handoff documentation. Use when user uploads .fig files, asks for "design specs", "component documentation", or "design-to-code handoff".
为什么好 点明干什么（分析 Figma、出研发交付文档）+ 什么时候用（上传 .fig、说 "design specs" 等具体短语）。文件类型也点了。
✓ GOOD · includes trigger phrases description: Manages Linear project workflows including sprint planning, task creation, and status tracking. Use when user mentions "sprint", "Linear tasks", "project planning", or asks to "create tickets".
为什么好 把触发短语明确列出来。Claude 看到 "sprint" / "create tickets" 这种关键词就会拉这个 Skill。

差的 description（反例三连）

✗ TOO VAGUE
description: Helps with projects.
差在哪
"projects" 太泛了，Claude 完全没法判断什么时候用。
✗ MISSING TRIGGERS
description: Creates sophisticated multi-page documentation systems.
差在哪
说了"干什么"，没说"什么时候用"。Claude 不知道用户哪句话该触发它。
✗ TOO TECHNICAL · NO USER TRIGGERS
description: Implements the Project entity model with hierarchical relationships.
差在哪
用了内部术语（Project entity model），用户不会这么说话。要用用户实际会说的话。

5.0 · CHOOSING YOUR APPROACH 切入方式 · Problem-first vs. Tool-first

Original · The Home Depot Analogy Think of it like Home Depot. You might walk in with a problem — "I need to fix a kitchen cabinet" — and an employee points you to the right tools. Or you might pick out a new drill and ask how to use it for your specific job.
中文导读 Home Depot（美国家居建材连锁店）类比：你可能带着问题进店——"我要修个橱柜"，店员给你指对应工具。也可能挑了个新电钻进店——问怎么用它做你的活。Skill 也是这两种切入：
PROBLEM-FIRST · 问题先行

"I need to set up a project workspace" → Your skill orchestrates the right MCP calls in the right sequence. Users describe outcomes; the skill handles the tools.

用户说"我要建个项目工作区" → Skill 自己编排该调哪些 MCP、什么顺序。用户描述结果，Skill 搞定工具。

TOOL-FIRST · 工具先行

"I have Notion MCP connected" → Your skill teaches Claude the optimal workflows and best practices. Users have access; the skill provides expertise.

用户说"我连了 Notion MCP" → Skill 教 Claude 怎么用它最有效。用户已有访问权，Skill 提供专家知识。

知道你的 Skill 是哪种切入，就更容易选对应的模式。

5.1 · PATTERN 1 Sequential workflow orchestration · 顺序工作流编排

Use when Your users need multi-step processes in a specific order.
中文导读 用户的流程步骤顺序固定、依赖明确——A 完成才能做 B，B 完成才能做 C。客户 onboarding、文档归档、订单处理都是典型。

示例结构（原文保留）

# Workflow: Onboard New Customer

## Step 1: Create Account
Call MCP tool: `create_customer`
Parameters: name, email, company

## Step 2: Setup Payment
Call MCP tool: `setup_payment_method`
Wait for: payment method verification

## Step 3: Create Subscription
Call MCP tool: `create_subscription`
Parameters: plan_id, customer_id (from Step 1)

## Step 4: Send Welcome Email
Call MCP tool: `send_email`
Template: welcome_email_template

关键技术

  • Explicit step ordering · 显式标号
  • Dependencies between steps · 步骤间的依赖（Step 3 用 Step 1 的输出）
  • Validation at each stage · 每步都校验
  • Rollback instructions for failures · 失败时的回滚指引

5.2 · PATTERN 2 Multi-MCP coordination · 多 MCP 协同

Use when Workflows span multiple services.
中文导读 流程要跨多个服务——比如"设计交付"涉及 Figma + Drive + Linear + Slack 四个 MCP。这种 Skill 关键在分阶段、阶段间传数据。

示例：设计到研发的交付（原文）

## Phase 1: Design Export (Figma MCP)
1. Export design assets from Figma
2. Generate design specifications
3. Create asset manifest

## Phase 2: Asset Storage (Drive MCP)
1. Create project folder in Drive
2. Upload all assets
3. Generate shareable links

## Phase 3: Task Creation (Linear MCP)
1. Create development tasks
2. Attach asset links to tasks
3. Assign to engineering team

## Phase 4: Notification (Slack MCP)
1. Post handoff summary to #engineering
2. Include asset links and task references

关键技术

  • Clear phase separation · 阶段分明
  • Data passing between MCPs · MCP 之间的数据传递（Phase 2 产出的链接给 Phase 3 用）
  • Validation before moving to next phase · 阶段切换前校验
  • Centralized error handling · 集中处理错误

5.3 · PATTERN 3 Iterative refinement · 迭代精修

Use when Output quality improves with iteration.
中文导读 不是一次成型的活——比如生成长报告、写复杂文档、做设计稿。需要"初稿 → 检查 → 精修 → 再检查"循环。

示例：报告生成（原文）

# Iterative Report Creation

## Initial Draft
1. Fetch data via MCP
2. Generate first draft report
3. Save to temporary file

## Quality Check
1. Run validation script: `scripts/check_report.py`
2. Identify issues:
   - Missing sections
   - Inconsistent formatting
   - Data validation errors

## Refinement Loop
1. Address each identified issue
2. Regenerate affected sections
3. Re-validate
4. Repeat until quality threshold met

## Finalization
1. Apply final formatting
2. Generate summary
3. Save final version

关键技术

  • Explicit quality criteria · 明确的"什么算够好"
  • Iterative improvement · 迭代改进
  • Validation scripts · 校验脚本（不靠语言判断）
  • Know when to stop iterating · 知道何时停（防止无限循环）

5.4 · PATTERN 4 Context-aware tool selection · 上下文感知的工具选择

Use when Same outcome, different tools depending on context.
中文导读 目标一样，但根据上下文条件挑不同的工具。文件存储就是典型——大文件、小文件、协作文档、临时文件应该走不同的存储。

示例：智能文件存储（原文）

# Smart File Storage

## Decision Tree
1. Check file type and size
2. Determine best storage location:
   - Large files (>10MB):     Use cloud storage MCP
   - Collaborative docs:     Use Notion/Docs MCP
   - Code files:             Use GitHub MCP
   - Temporary files:        Use local storage

## Execute Storage
Based on decision:
- Call appropriate MCP tool
- Apply service-specific metadata
- Generate access link

## Provide Context to User
Explain why that storage was chosen

关键技术

  • Clear decision criteria · 决策规则清晰
  • Fallback options · 各种情况都有兜底
  • Transparency about choices · 跟用户解释为什么选这个

5.5 · PATTERN 5 Domain-specific intelligence · 领域专家知识

Use when Your skill adds specialized knowledge beyond tool access.
中文导读 Skill 不只是"用工具"，还在嵌入领域专家的判断逻辑——比如合规审计、医疗筛查、法律审核。这类 Skill 的价值不在"调对工具"，而在"行动前的领域判断"。

示例：支付合规处理（原文）

# Payment Processing with Compliance

## Before Processing (Compliance Check)
1. Fetch transaction details via MCP
2. Apply compliance rules:
   - Check sanctions lists
   - Verify jurisdiction allowances
   - Assess risk level
3. Document compliance decision

## Processing
IF compliance passed:
  - Call payment processing MCP tool
  - Apply appropriate fraud checks
  - Process transaction
ELSE:
  - Flag for review
  - Create compliance case

## Audit Trail
- Log all compliance checks
- Record processing decisions
- Generate audit report

关键技术

  • Domain expertise embedded in logic · 领域知识嵌进决策逻辑
  • Compliance before action · 合规检查在动作前
  • Comprehensive documentation · 全程记录
  • Clear governance · 治理规则清晰

5.6 · TROUBLESHOOTING — UPLOAD 排错 · 上传失败

Error: "Could not find SKILL.md in uploaded folder"

CAUSE
File not named exactly SKILL.md
SOLUTION
改名为精确的 SKILL.md（大小写敏感），用 ls -la 确认文件名。

Error: "Invalid frontmatter"

YAML 格式错误的三种常见写法（原文 + 注释） 
# Wrong — missing delimiters（缺 --- 分隔符）
name: my-skill
description: Does things

# Wrong — unclosed quotes（引号没闭合）
name: my-skill
description: "Does things

# Correct ✓
---
name: my-skill
description: Does things
---

Error: "Invalid skill name"

# Wrong — has spaces and capitals
name: My Cool Skill

# Correct ✓
name: my-cool-skill

5.7 · TROUBLESHOOTING — TRIGGERING 排错 · 不触发 / 过触发

Skill doesn't trigger · 不触发

Fix Revise your description field.

Quick checklist:
• Is it too generic? ("Helps with projects" won't work)
• Does it include trigger phrases users would actually say?
• Does it mention relevant file types if applicable?
中文导读 改 description（参考第 2 章 2.6）。三件事自查：太泛了吗？包含用户真会说的短语吗？涉及的文件类型点明了吗？

调试技巧（debugging approach）

Debugging technique Ask Claude: "When would you use the [skill name] skill?" Claude will quote the description back. Adjust based on what's missing.
中文导读 直接问 Claude：「你什么时候会用 [skill name] 这个 Skill？」Claude 会把它从 description 里读到的理解复述给你听——你就知道少了什么。

Skill triggers too often · 过触发

SOLUTION 1 · 加负面触发词
description: Advanced data analysis for CSV files. Use for statistical modeling, regression, clustering. Do NOT use for simple data exploration (use data-viz skill instead).
SOLUTION 2 · 更具体
# Too broad
description: Processes documents

# More specific ✓
description: Processes PDF legal documents for contract review

Check · 检查项What to do · 怎么做

5.9 · TROUBLESHOOTING — INSTRUCTIONS 排错 · 指令不被遵循

Symptom Skill loads but Claude doesn't follow instructions.

原因 1 · Instructions too verbose · 啰嗦

SYMPTOM · 症状

SKILL.md 太长、太啰嗦、关键指令被埋没

FIX · 解决

用 bullet / 编号列表 · 详细文档挪到 references/ · 保持 SKILL.md 简洁

SYMPTOM

关键约束写在 SKILL.md 中段，Claude 没注意到

FIX

关键指令放最顶部 · 用 ## Important 或 ## Critical 标题 · 重要点可以重复

✗ BAD

差在哪

✓ GOOD

好在哪

• Quality is more important than speed

安全说明（Security Notes）

✓ ALLOWED · 允许
