1.1 · WHAT IS A SKILL 一个 Skill 长什么样？

Skill 不是模型、不是插件、不是 prompt 模板——它就是一个文件夹。文件夹里有一个主说明文件，另外可能还塞一些脚本、参考资料、模板素材。Claude 读懂这个文件夹的内容，就具备了对应的能力。

官方对 Skill 的定义只有一句话：「a set of instructions packaged as a simple folder」——一组打包成简单文件夹的说明书。这个定义朴素到值得记住——它说明 Skill 没有任何"魔法"，只是一份组织得当的文件集。

标准结构

# 一个典型的 Skill 文件夹长这样
your-skill-name/
├── SKILL.md              # 必有 · 主说明文件（YAML 头 + Markdown 正文）
├── scripts/              # 可选 · 可执行代码（Python / Bash 之类）
│   ├── process_data.py
│   └── validate.sh
├── references/           # 可选 · Claude 用得到时才加载的扩展文档
│   ├── api-guide.md
│   └── examples/
└── assets/               # 可选 · 输出时要用的模板、字体、图标等素材
    └── report-template.md

四种东西：

  • SKILL.md · 唯一必需的文件 —— 主说明书。文件名严格大小写，写错（skill.md / SKILL.MD）一律不认。
  • scripts/ · 脚本目录 —— 把"用语言描述很啰嗦、用代码三行搞定"的活塞进来。比如格式校验、数据清洗、调外部 API。
  • references/ · 参考资料目录 —— 详细文档、范例、API 说明。Claude 需要时才会去翻，不主动加载。
  • assets/ · 素材目录 —— 文档模板、字体、品牌色 SVG、报告骨架等"生成时引用的成品材料"。

命名约定（这是规矩，不是建议）

✓ DO · 应该这样
  • notion-project-setup
  • figma-design-export
  • quarterly-report-builder

短横线分词、全部小写。术语叫 kebab-case（烤串命名，因为短横线像烤串签）。

✗ DON'T · 不要这样
  • Notion Project Setup （有空格）
  • notion_project_setup （下划线）
  • NotionProjectSetup （大驼峰）
  • 名字里含 claude 或 anthropic （系统保留）
特别提醒 · SKILL 文件夹里别放 README.md Skill 内部不要放 README.md——所有"给 Claude 看"的内容都应该在 SKILL.md 或 references/ 里。但如果你把这个 Skill 发布到 GitHub 仓库，仓库根目录还是应该放 README 给人类访问者看——这是仓库的 README，不在 Skill 文件夹里面。

1.2 · PROGRESSIVE DISCLOSURE 核心设计原则一 · 渐进披露

Progressive Disclosure（渐进披露 / 按需展开）是 Skill 最重要的设计原则，它解决一个简单问题：不能把所有 Skill 的全部内容都塞进 Claude 的上下文，否则光是"看完所有手册"就把 token 耗光了。

所以 Skill 采用了"按需加载"——内容被切成三层，离 Claude 上下文从近到远：

01
LEVEL 1 · YAML FRONTMATTER · 一直加载

头部元数据 ——「这个 Skill 是干嘛的，什么时候用」

就是 SKILL.md 顶部那几行 --- 包起来的配置。永远在 Claude 的系统 prompt 里。它的任务是让 Claude 在看到用户消息时判断要不要把这个 Skill 拉进来——所以信息必须精炼，描述要包含触发短语。

02
LEVEL 2 · SKILL.md BODY · 判断相关时加载

正文说明 ——「具体怎么做」

Claude 判断"这次任务跟某个 Skill 有关"后，才把 SKILL.md 的正文加载进来。里面是完整的步骤、注意事项、示例。不在第一层是因为：用不到的时候不该占 token。

03
LEVEL 3 · LINKED FILES · 探索式加载

附加文件 ——「碰到具体问题时才翻」

放在 references/ 等子目录里的扩展文档。Claude 在执行中真的需要时（比如查某个 API 的细节）才会主动去读对应文件。最大的好处：一个 Skill 可以挂大量参考资料而不占基础上下文。

Progressive disclosure minimizes token usage
while maintaining specialized expertise.

用一个生活化的类比：第一层像电梯里的楼层目录（每次进电梯都看一眼）；第二层像每层走廊上的科室指引（到这层才看）；第三层像各科室门口的详细公告板（真要办这个科的事才进去读）。三层是同一份信息的不同粒度，Claude 按需深入。

1.3 · COMPOSABILITY 核心设计原则二 · 可组合

Claude 可以同时加载多个 Skill。你写自己的 Skill 时，不能假设它是唯一一个被启用的。

这听起来是个小细节，但写法会差很多。举个例子：

✗ 假设自己是唯一 Skill 的写法

「所有用户输入都用 Markdown 输出。」

问题：另一个生成 PDF 的 Skill 也开着，两边互相打架。

✓ 假设要跟别人共存的写法

「当用户要求本 Skill 处理的报告类型时，输出 Markdown 格式。」

限定了适用范围，其他 Skill 各管各的。

核心心法是：你的 Skill 只对它自己负责的场景说话，别越权改写"通用规则"。

1.4 · PORTABILITY 核心设计原则三 · 可移植

同一个 Skill 在 Claude.ai 网页版、Claude Code 命令行、和 API 上的行为完全一致。写一次，三个地方都能用，不用改。

这点的代价是：Skill 自身不能依赖某个特定环境独有的工具。如果你的 Skill 必须用某个特殊系统包或网络访问，要在 frontmatter 的 compatibility（兼容性）字段里声明出来，这样使用者能预判它在自己的环境里能不能跑。

可移植性意味着 Skill 是跨产品的资产——你给团队写的一个 Skill，可以同时被网页用户、命令行开发者、API 调用者享用，不用维护三份。

1.5 · SKILLS + MCP Skills 和 MCP 是搭档，不是替代关系

💡 如果你不做 MCP · YOU MAY SKIP 本节是写给已经在做 MCP server（连接器后端）的人的。如果你只想做独立 Skill、不接外部服务，可以跳到第 2 章；后面有需要再回来读这节。

MCP（Model Context Protocol，模型上下文协议）是 Claude 连接外部服务（Notion、Linear、Asana 等等）的协议层。它解决"Claude 能不能用上你的服务"的问题。Skill 解决的是另一个问题——"Claude 该怎么用好你的服务"。

厨房类比

🍳
MCP
专业厨房 ——
炉灶、刀具、各种食材，
给厨师"用得着的家伙"
+
📖
Skill
食谱 ——
一步步告诉你怎么用
这些工具做出菜来

两者放一起，用户才能在"不需要自己琢磨步骤"的情况下完成复杂任务。光有 MCP 没 Skill，用户上来面对一堆工具，不知道哪步先哪步后；光有 Skill 没 MCP，等于光有食谱没厨房。

角色对照表

对比维度 MCP · 连接层（Connectivity） Skill · 知识层（Knowledge）
提供什么 把 Claude 连到你的服务（Notion / Asana / Linear …） 教 Claude 怎么有效地用你的服务
给 Claude 的能力 实时拉取数据 + 调用工具 捕获工作流程 + 沉淀最佳实践
回答的问题 Claude 能做什么 Claude 该怎么做

给 MCP 用户加 Skill 的差别

✗ 只有 MCP 没 Skill
  • 用户连上了你的 MCP，但不知道接下来该干嘛
  • 支持工单不断："你这个集成里怎么做 X？"
  • 每次会话都从零开始
  • 不同用户 prompt 写法不同，结果不一致
  • 用户觉得是连接器有问题，其实是缺工作流指引
✓ MCP + 配套 Skill
  • 预设好的工作流在需要时自动激活
  • 工具调用一致、可靠
  • 最佳实践内嵌在每一次交互里
  • 用户上手成本显著降低

1.6 · TAKEAWAYS 小结 · 这一章给你的 4 个判断

  1. Skill 是一个文件夹——里面必有 SKILL.md，可选 scripts/ / references/ / assets/。任何复杂度都从"加上一个文件"开始扩展。
  2. 三层渐进披露是 Skill 不爆 token 的根本——头部 YAML 永久驻留判触发，正文按需加载，链接文件深度探索。写 Skill 时要时刻问自己：这段内容该放哪一层？
  3. 可组合 + 可移植意味着写法上要克制——别假设自己是唯一 Skill、别依赖某个产品独有的能力（除非声明 compatibility）。
  4. Skill 和 MCP 是知识层 + 连接层的搭档。如果你已经有 MCP，加一个配套 Skill 是放大 MCP 价值的最快办法；如果你只做独立 Skill，前面三个原则就是全部基础，可以直接进入第 2 章规划与设计。
